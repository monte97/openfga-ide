# User Documentation — Design Spec

**Date:** 2026-04-08
**Project:** openfga-viewer
**Scope:** Bilingual (EN + IT) user documentation with Playwright-generated screenshots

---

## Overview

Create end-user documentation for openfga-viewer covering all functional areas of the tool. Documentation is bilingual (English and Italian), stored as Markdown files in the repository, and accompanied by reproducible screenshots generated via a dedicated Playwright script using the existing demo fixture.

## Page Content Outline

### README.md (root)
- What is openfga-viewer
- Prerequisites (Node.js, Docker, an OpenFGA instance)
- Quick install (`npm install`, `npm run dev`)
- Link to full English docs (`docs/en/01-getting-started.md`)
- Link to Italian docs (`docs/it/01-getting-started.md`)

### 01-getting-started
- System requirements
- Installation steps (clone → install → configure environment)
- Starting the app (dev mode and Docker)
- First connection to an OpenFGA instance
- Loading the demo dataset (reference to `demo/demo-document-sharing.json`)

### 02-connection-and-stores
- Connection configuration (URL, Store ID, API key optional)
- Testing the connection
- Selecting an active store from the dropdown (StoreSelector)
- Store Admin: creating, deleting stores
- Connection status indicator in the header

### 03-model-tuples-queries
- **Model Viewer:** DSL view with syntax highlighting; Graph view with interactive nodes
- **Tuple Management:** table with filtering and pagination; adding and deleting tuples
- **Query Console:** Check query (green/red result + why explanation); List Objects; List Users; Expand tree

### 04-relationship-graph
- Navigating the canvas (pan, zoom)
- Filtering by type and relation
- Selecting a node to open the inspector panel
- Reading the inspector panel (related tuples, relations)

### 05-import-export
- Exporting store data (model + tuples) as JSON
- Importing from a JSON file (validation preview)
- Using export/import for backup and restore

### 06-test-suites
- What a test suite is (definition, fixture, test cases)
- Suite list: creating a new suite, viewing last run status
- Deleting a suite
- Understanding suite metadata (name, description, tags)

### 07-suite-editor
- **Form editor:** adding groups, adding test cases (user / relation / object / expected)
- **JSON editor:** switching to raw JSON, editing the full definition
- Dual-mode sync (form ↔ JSON without data loss)
- **Fixture editor:** defining model and tuples inline; importing from current store

### 08-test-execution
- Triggering a test run
- Phase timeline (provisioning → running → cleanup)
- Polling and circuit breaker (what happens on network issues)
- Reading results: summary stats, per-test pass/fail, expected vs actual
- Last run badge on the suite list

## Screenshot Pipeline

### Script: `docs/scripts/capture-screenshots.ts`

**Dependencies:** Playwright (already installed in the project as `@playwright/test`)

**Prerequisites for running:**
- Frontend running at `http://localhost:5173`
- Backend running at `http://localhost:3000`
- A local OpenFGA instance running at `http://localhost:8080`

**What the script does:**
1. Connects to the running app via Playwright (Chromium)
2. Navigates to each section of the app in sequence
3. Sets up demo state via the backend API (imports `demo/demo-document-sharing.json`)
4. Captures a screenshot for each defined shot
5. Saves all screenshots to `docs/assets/screenshots/`
6. Prints a summary of captured files

**npm script (package.json root):**
```json
"docs:screenshots": "<runner> docs/scripts/capture-screenshots.ts"
```
Runner to be determined during implementation (options: `tsx`, `ts-node`, or `npx playwright test --grep screenshots`).

## Bilingual Strategy

- `docs/en/` and `docs/it/` are fully independent — no sync automation
- Screenshots are shared: both languages reference the same `docs/assets/screenshots/` paths
- The root `README.md` is in English with a prominent link to `docs/it/01-getting-started.md`
- Content is written natively in each language (not translated mechanically)

---

## Out of Scope

- VitePress or any static site generator
- Search functionality
- Automated translation sync
- API reference documentation (internal tool, no public API)
- Versioning of documentation

---

## Implementation Notes

- The Playwright script must handle the case where the app has no stores configured yet (fresh state). It should create a store via the backend API before capturing screenshots.
- Screenshots should use a fixed viewport (1440×900) for consistency.
- The demo fixture `demo/demo-document-sharing.json` contains a document-sharing authorization model — all screenshots will show this model.
- The script should be idempotent: re-running it overwrites existing screenshots without error.
