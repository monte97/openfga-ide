### 2. `document.createElement` mock â†’ infinite recursion in view tests

In Story 6.1, the initial approach to mocking `document.createElement` in `ImportExport.vue` tests triggered infinite recursion. The correct fix â€” mocking `@/stores/importExport` instead â€” was found, but required debugging time.

## Root Cause Analysis

**Why do story files go unupdated?**
The dev agent marks `sprint-status.yaml` as `done` and considers the story complete. Updating the story file itself (task checkboxes, completion notes, file list) is a separate step with no automated enforcement â€” it gets skipped under time pressure or forgotten.

## Lessons Learned

1. **Story file discipline is a completion gate, not an afterthought.** Sprint-status `done` should not be set until the story file reflects reality. The spec file is the source of truth for what was built and how.

2. **`document.createElement` in view tests: mock the store, not the DOM.** When a component triggers a download via Blob URL internally, testing it by mocking `document.createElement` causes infinite recursion. The correct approach is to mock the Pinia store action that contains the download logic.

3. **Accumulated patterns compound.** By Epic 6, the codebase had enough well-documented prior patterns (pagination, toasts, dialogs, routers) that both stories were implemented with minimal re-investigation. The investment in dev notes across Epics 1-5 paid a clear dividend.

4. **3/3 action item follow-through is achievable.** When action items are specific, codified in CLAUDE.md, and surfaced in story dev notes, they get applied. Vague action items ("be more careful about X") get ignored; concrete ones ("use `vi.mock` instead of `createTestingPinia`") get followed.

## Project-Level Reflection

Epic 6 closes the openfga-viewer MVP roadmap. The full feature set â€” store management, model visualization, tuple management, permission queries, relationship graph, and import/export â€” has been delivered across 6 epics and 18 stories.

The BMad workflow, CLAUDE.md conventions, and accumulated story dev notes provided consistent context across all epics. Key conventions that emerged and should be carried into future projects:

- `vi.mock` for Pinia stores (not `@pinia/testing`)
- Reactive mock pattern for `watch` tests
- `mock.results.at(-1)` for Pinia store instance isolation
- Pinia reactive Set: always replace reference, never mutate in place
- `v-show` + `nextTick` for CSS transitions (not `v-if`)
- ESM `.js` extensions on all backend imports
- Story file updates as part of completion, not an afterthought
