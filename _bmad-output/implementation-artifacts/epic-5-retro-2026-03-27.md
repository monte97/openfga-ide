## What Went Well

**Pattern reuse was excellent.**
Dagre layout, Vue Flow dark CSS variables, TypeBadge color system, and edge deduplication guard were all inherited cleanly from Story 2.3. No duplicate implementations. The investment in establishing these patterns in Epic 2 paid off in Epic 5.

**Pinia reactive Set pattern applied correctly.**
The `toggleTypeVisibility` action correctly replaces the `Set` reference instead of mutating it in place — a non-obvious Vue 3 reactivity requirement that was handled proactively.

**Architectural separation of concerns.**
`GraphNodeDetail` in `components/graph/` was kept distinct from the `components/model/GraphNodeDetail.vue` from Story 2.3. The two components serve different data shapes and the boundary was respected.

**`v-show` + `nextTick` animation fix applied.**
Story 2.3's lesson about `v-if` preventing CSS transitions was carried forward correctly into Story 5.2's inspector panel.

## What Didn't Go Well

### 1. Architectural decision deferred in 5-1, reversed in 5-2

Story 5-1 opted to skip creating `relationshipGraph.ts`, calling it "optional." Story 5-2 had to create it anyway because filter state and selection state require a proper store. The dependency was foreseeable: the story dev notes explicitly stated "a thin coordinator store provides a clean abstraction point."

**Root cause:** Premature optimization — "we don't need it yet" reasoning when the next story makes it inevitable.

### 2. Recurring reactive/event test coverage gaps (both stories)

Both stories were submitted for code review with missing tests. Total: 5 tests added post-implementation.

| Story | Missing tests |
|-------|--------------|
| 5-1 | `watch(storeId)` re-fetch; `loadGraph()` action integration |
| 5-2 | Toggle-back (re-check); "No relationships" empty state; `@node-click` integration |

All five are boundary conditions or reactive behaviors — not core happy paths. The happy path unit tests were written; the "what happens when state changes" scenarios were consistently missed.

### 3. Dev notes referenced `@pinia/testing` (not installed)

Story 5-2 dev notes suggested `createTestingPinia` as the test pattern. The project uses `vi.mock` consistently because `@pinia/testing` is not installed. This creates confusion when dev agents follow the dev notes instead of the established project pattern.

---

## Root Cause Analysis

**Why do reactive/event tests get missed?**
1. They require a different test setup (reactive mocks, `nextTick`) and are harder to write.
2. Story task checklists specify test cases for primary ACs but don't enumerate boundary/reactive scenarios explicitly. The dev agent writes what's specified; unspecified scenarios are skipped.

## Lessons Learned

1. **Foreseeable dependencies should not be deferred.** If a story's dev notes say "X provides a clean abstraction point", and the next story in the same epic clearly needs X, don't skip X.

2. **Test checklists must be explicit about reactive behaviors.** "Co-locate test file" is not a test specification. Enumerate the scenarios, especially: `watch` callbacks, `onMounted` + `onUnmounted` side effects, keyboard/mouse event handlers, and store action call sequences.

3. **`mock.results.at(-1)` for Pinia store isolation.** When a Pinia store calls another Pinia store internally, and you need to reference the mock instance created during a specific test, use `.mock.results.at(-1)!.value` rather than `.mock.results[0].value` to avoid cross-test contamination.

4. **Pattern anchors from earlier epics pay compound dividends.** The Vue Flow, dagre, and TypeBadge patterns established in Epic 2 required zero re-invention in Epic 5. The upfront pattern investment was worthwhile.
