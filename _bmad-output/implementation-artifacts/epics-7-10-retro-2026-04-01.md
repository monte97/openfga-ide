## What Didn't Go Well

### 1. Debounce not included in initial implementations

Both `SuiteEditor.vue` (onJsonChange) and `FixtureEditor.vue` (onJsonChange) launched concurrent `saveDefinition` API calls on every keystroke. Debounce was deferred during dev — but since both components were keyboard-driven editors, it should have been in scope from the start.

**Root cause:** Dev notes flagged debounce as "future" without checking whether the AC required save-on-change behavior (it did). Debounce is mandatory for any text editor that triggers API calls.

### 2. Suite-switch state not reset in initial pass

`editorMode`, `expandedGroupIds`, and the fixture validation banner all persisted across suite navigation. Three separate pieces of state, three separate bugs, all with the same root cause.

**Root cause:** The `watch(() => props.suite.id)` handler was added for fetch and deselect, but editor view state was considered "local" and not included in the reset. Any navigation-triggered state reset must enumerate all relevant local state explicitly.

### 3. View directly mutating store state

`SuiteEditor.vue` did `suiteStore.activeSuite = { ...suiteStore.activeSuite, definition: parsed }` — a direct mutation of store state from a view. Pinia actions exist precisely to encapsulate this. The `patchDefinition` action was only added during the post-sprint cleanup.

**Root cause:** No explicit convention preventing direct store field assignment from views. The store already had fine-grained actions (`addGroup`, `updateTestCase`, etc.) for everything else; `patchDefinition` should have been designed alongside them.

### 4. Missing null guards on API-sourced data

`useAutocompleteOptions` iterated over `tupleStore.tuples` without guarding against `tuple.key.user/relation/object` being null/undefined. Malformed backend responses would push `{ value: undefined, label: undefined }` into `SearchableSelect`.

**Root cause:** Trust in backend data shape without defensive null guards at the composable boundary. External API data should always be guarded at the point of consumption.

### 5. Unnecessary saves on blur

`TestCaseForm.onDescriptionBlur` and `onTagsBlur` fired `save()` even when the value was unchanged, generating redundant PUT requests on every field blur.

**Root cause:** Blur handlers were written as "always save" without dirty-checking against the current prop value. Blur handlers that persist optional fields need a dirty check.

## Root Cause Analysis

**Why do these bugs cluster?**
Epics 7–10 involved more complex stateful components than Epics 1–6: dual-mode editors, debounced persistence, navigation-triggered resets, and composables consuming external data. The conventions established for simpler store/component patterns (vi.mock, reactive Set, v-show+nextTick) did not cover these new complexity domains.

## Lessons Learned

1. **Debounce is mandatory for editors that save on change, not optional.** Any component where the user types freely and the result is persisted to an API must debounce those calls. Include it in the initial story scope, not as a deferred enhancement.

2. **Navigation-triggered state resets must be exhaustive.** When `suite.id` changes, all editor state should reset: mode, selection, expanded groups, validation banners. The watch handler is the single point of reset — enumerate everything there.

3. **Store actions own all mutations.** Views should not assign to store fields directly. If no action exists for the mutation you need, add one. `patchDefinition`, `clearExpandedGroups` — these are cheap to add and make testing, debugging, and future refactoring easier.

4. **External API data is untrusted at the composable boundary.** Even if the backend is our own, defensive null guards in composables prevent malformed responses from propagating silently into the UI.

5. **`deferred-work.md` as structured backlog enables post-sprint cleanup.** Recording deferred items with context (which review, what the risk is) made a single focused cleanup session highly productive. Without this structure, the bugs would have required archaeology to rediscover.

6. **Post-implementation refinement sessions are a legitimate delivery phase.** The bugs fixed on 2026-04-01 were not caught during initial development. Dedicating time after all stories are done — before declaring the epic set complete — is a valid and valuable practice.

## Project-Level Reflection

Epics 7–10 close the full openfga-viewer feature set. The product now covers: store management, model visualization, tuple management, permission queries, relationship graph, import/export/backup, and a complete test suite authoring and execution system.

The BMad workflow held up well across 20 total stories (Epics 1–10). The conventions accumulated in CLAUDE.md — `vi.mock` pattern, reactive Set, `mock.results.at(-1)`, `v-show`+`nextTick`, `document.createElement` mock avoidance — provided consistent context and prevented entire classes of bugs from recurring.

Conventions to carry into the next project (if any):

- `vi.mock` for Pinia stores, never `createTestingPinia`
- Reactive mock pattern for `watch` tests
- `mock.results.at(-1)` for Pinia store instance isolation
- Pinia reactive Set: always replace reference, never mutate in place
- `v-show` + `nextTick` for CSS transitions (not `v-if`)
- Debounce required for any editor persisting to an API
- Store actions for all mutations from views
- Null guards at all external API data boundaries
- Dirty-check before blur saves on optional fields
- Navigation `watch` handlers must reset all local editor state
