# Story 2.1: Backend Model Endpoint and DSL Conversion

Status: done

## Story

As a developer,
I want the backend to serve the authorization model in both JSON and DSL formats,
so that the frontend can display the model in the user's preferred format.

## Acceptance Criteria

1. **Given** a store with an authorization model is selected **When** I send `GET /api/stores/:storeId/model` **Then** I receive `{ "json": <model_json>, "dsl": "<model_dsl>", "authorizationModelId": "<id>" }` with the DSL generated by `@openfga/syntax-transformer`

2. **Given** a store with no authorization model **When** I send `GET /api/stores/:storeId/model` **Then** I receive `{ "json": null, "dsl": null, "authorizationModelId": null }`

3. **Given** the `@openfga/syntax-transformer` package **When** the backend starts **Then** the pinned version is loaded and validated, and DSL conversion is tested with a known input at startup (smoke test)

4. **Given** the syntax-transformer fails to convert a model **When** the backend processes the model response **Then** it returns the JSON model with `"dsl": null` and logs a warning with Pino, rather than failing the entire request

5. **Given** a `storeId` that does not exist **When** I send `GET /api/stores/:storeId/model` **Then** I receive a `404` with `{ "error": "Store not found" }`

6. **Given** the OpenFGA instance is unreachable **When** I send `GET /api/stores/:storeId/model` **Then** I receive a `500` with `{ "error": "Failed to fetch model", "details": "<message>" }`

### Architecture Compliance

- **Thin passthrough route:** `routes/model.ts` receives the HTTP request, calls `modelService.getModel(storeId)`, returns JSON. No business logic in the route. [Source: architecture.md#Architectural Boundaries]
- **Service layer owns transformation:** `model-service.ts` fetches from OpenFGA via `openfga-client` and converts to DSL. The route does NOT call the transformer directly. [Source: architecture.md#Service Boundaries]
- **Single OpenFGA contact point:** ALL OpenFGA HTTP calls go through `openfga-client.ts`. No direct `fetch`/`axios` calls in `model-service.ts`. [Source: architecture.md#Architectural Boundaries]
- **Error envelope:** All backend errors return `{ error: string, details?: any }` ŌĆö enforced by global error handler. [Source: architecture.md#Format Patterns]

### Critical Technical Details

**OpenFGA API endpoint for models:**
```
GET /stores/{storeId}/authorization-models
```
Returns:
```json
{
  "authorization_models": [
    {
      "id": "01FMKABCDEFGH",
      "schema_version": "1.1",
      "type_definitions": [...],
      "conditions": {}
    }
  ],
  "continuation_token": ""
}
```
- Models are returned newest-first. Use `page_size=1` to get only the latest: `/stores/{storeId}/authorization-models?page_size=1`
- If no model exists: `"authorization_models": []` with 200 status (not a 404)
- The `id` field on the model object is the `authorizationModelId`

**`@openfga/syntax-transformer` usage:**
```typescript
import { transformJSONStringToDSL } from '@openfga/syntax-transformer'
// OR (check actual package exports ŌĆö may differ)
import { OpenFgaDslTransformer } from '@openfga/syntax-transformer'

// The function takes the model JSON as a STRING, not an object
const dsl = transformJSONStringToDSL(JSON.stringify(model))
```
IMPORTANT: Verify the exact export name by checking the package's `index.d.ts` or README after install. The package may export `transformJSONStringToDSL` or a class with a static method. Adapt accordingly.

### What NOT to Do

- **Do NOT implement any frontend code** ŌĆö Story 2.1 is backend-only. The `ModelViewer.vue`, `ModelDslView.vue`, `ModelGraphView.vue`, and `frontend/src/stores/model.ts` are all built in Stories 2.2 and 2.3.
- **Do NOT use `@openfga/sdk`** ŌĆö the project deliberately uses `openfga-client.ts` as a thin HTTP client. Never install the official SDK.
- **Do NOT fetch all pages of models** ŌĆö use `page_size=1` to get only the latest model. The OpenFGA API returns models newest-first, so index 0 is always the active model.
- **Do NOT call the transformer at the route level** ŌĆö DSL conversion belongs in `model-service.ts`, not in the route handler.
- **Do NOT throw from the transformer catch block** ŌĆö AC#4 explicitly requires graceful degradation: return `dsl: null` with a warning log, keep the request successful.
- **Do NOT create a new HTTP client** ŌĆö use `openfgaClient` from `openfga-client.ts` exclusively.
- **Do NOT skip the startup validation** ŌĆö NFR12 requires the transformer to be validated at startup. A broken transformer should prevent the server from starting (`process.exit(1)`).
- **Do NOT hardcode the storeId** ŌĆö it comes from `req.params.storeId` passed by the route param. The `openfga-client.ts` may already have a storeId concept from Story 1.2/1.5 ŌĆö check whether `getModel` should use the client's built-in storeId or always pass it explicitly. The GET endpoint path includes the storeId explicitly, so pass it through.
- **Do NOT use `any` type** ŌĆö TypeScript strict mode is enforced. Define proper types in `backend/src/types/openfga.ts`.
- **Do NOT use CommonJS `require()`** ŌĆö backend is ESM (`"type": "module"`). All imports must use `import`.

### Debug Log References

1. **`@openfga/syntax-transformer` CJS/ESM interop**: The package is CJS and exports a nested object. Named ESM import `import { transformJSONStringToDSL } from '@openfga/syntax-transformer'` fails at runtime. Fix: use default import and access `syntaxTransformer.transformer.transformJSONStringToDSL`. Cast required: `(syntaxTransformer as unknown as { transformer: { transformJSONStringToDSL: (json: string) => string } }).transformer`.

2. **`@openfga/syntax-transformer` requires `@openfga/sdk` as undeclared peer**: Despite v0.2.1 package.json listing no peer dependencies, the compiled dist imports `@openfga/sdk/dist/utils`. Fixed by installing `@openfga/sdk@0.9.3` as an explicit dependency. This is acceptable ŌĆö `@openfga/sdk` provides type utilities used by the transformer, not API call logic.

3. **Vitest picking up `.stryker-tmp/` sandbox test files**: Vitest's default glob matched test files in `.stryker-tmp/sandbox-*/src/routes/`, causing EADDRINUSE port conflicts when both real and sandbox server tests tried to bind the same port. Fixed by creating `backend/vitest.config.ts` with `exclude: ['.stryker-tmp/**', 'node_modules/**']`. Test count corrected from 77 ŌåÆ 43.

4. **TypeScript TS2352 double-cast in mocks**: Vitest's updated types require `as unknown as { ... }` double-cast when asserting mock function types. Applied to all new test files and patched pre-existing `stores.test.ts` and `store-service.test.ts`.

5. **`req.params['storeId']` typed as `string | string[]`**: Express types `req.params` values as `string | string[]`. Fixed with `as string` cast since `storeId` is always a single path segment.
