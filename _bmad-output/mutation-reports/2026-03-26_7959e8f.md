## Azioni consigliate

### Priorità alta
1. **Eliminare `src/stores/counter.ts`** — scaffolding inutilizzato, inquina le metriche
2. **Verificare URL nelle chiamate API** — aggiungere `expect(fetchMock).toHaveBeenCalledWith('/api/stores', ...)` nei test degli store
3. **Verificare request body** — `expect(fetchMock).toHaveBeenCalledWith(expect.any(String), expect.objectContaining({ body: JSON.stringify({ name }) }))`

### Priorità media
4. **Testare loading state** — verificare `store.loading === true` durante le chiamate async (con `vi.useFakeTimers` o promise non ancora risolte)
5. **Testare error reset** — verificare che `error` si azzeri prima di ogni chiamata

### Priorità bassa
6. **Backend config.ts** — testare i default con env vars non impostati (minor valore)
7. **error-handler.ts log payload** — testare il contenuto dei log Pino (overhead alto per valore basso)

## Note metodologiche

- **Scope frontend limitato a `stores/`**: i composables (`useApi.ts`) sono stati esclusi perché usano `fetch` direttamente e generano molti timeout con Stryker (213 mutanti, >20 min stimati). Causa: il network mock non è attivo nel sandbox Stryker. Soluzione futura: configurare `setupFiles` di Vitest per stub fetch anche in modalità Stryker.
- **counter.ts**: incluso per glob `src/stores/**/*.ts` — rimuovere il file risolve il problema a monte.
- **HTML report interattivo**: disponibile in `_bmad-output/mutation-reports/backend/index.html` e `frontend/index.html`
